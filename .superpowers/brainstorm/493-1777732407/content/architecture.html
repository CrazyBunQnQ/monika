架构变更：单引擎 → 多引擎

启动时初始化所有已配置的供应商，存入 map，切换供应商即切换 map key

当前：单引擎
App {
  provider: ProviderEngine // 唯一
  model: string
  cfg: Config
}

GetModels() → one engine
SendMessage(..., model)
目标：多引擎
App {
  providers: map[string]ProviderEngine
  model: string
  cfg: Config
}

GetProviders() → []string
GetModels(providerID) → one engine
SendMessage(..., provider, model)

后端改动清单

1. bootstrap/provider.go

InitProvider() 遍历 cfg.ModelProviders 的每个 key，逐个初始化引擎，返回 map[string]ProviderEngine 而非单个。

2. internal/api/app.go

新增 GetProviders() 返回已配置的供应商 ID 列表。
GetModels(providerID) 接受参数定位引擎。
SendMessage(..., provider) 接受供应商参数，传给 agent loop。

3. internal/agent/agent_loop.go

LoopOption 增加 WithProvider()，ChatRequest.Provider 填入该值。

4. 配置持久化

SendMessage 成功后更新 cfg.ModelProvider 和 cfg.Model 写入 ~/.monika/config.yaml。

前端改动清单

1. Zustand store

新增: availableProviders: string[], selectedProvider: string, loadProviders(), setSelectedProvider()。
修改: loadModels(providerID) 按供应商加载模型列表。

2. ModelPicker 组件（新建）

替代 ChatInputToolbar 中的单一下拉框。按钮显示供应商图标+模型名，点击弹出分组列表。

3. ChatInputToolbar 改动

用 ModelPicker 替换原有的 <select> 元素。

4. Wails bindings

重新生成以反映 API 变更。

这个架构是否合理？确认后继续细化 ModelPicker 的交互设计。
