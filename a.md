## 1 背景与目标

### 1.1 初始想法

1. 用户可以上传跑团脚本
2. 大模型作为 KP 主持游戏，应该允许 player 发散去玩，同时又能推动游戏进行
3. 允许多 player 同时跑团
4. 把规则书作为知识库
5. 大模型应该能记住每个人的角色卡

### 1.2 范围（In Scope）

- v1 目标：以 CoC 7e 为默认规则系统，提供可玩的跑团闭环（可调查、可检定、可战斗、可追逐、可规则问答）
- 交互形态：自然语言优先 + 少量命令快捷键（TRPG-only 门禁）；KP 从语义中提取“意图/行动/机制触发”，并把机制结果以稳定结构输出
- 运行形态：支持长 Session 的记录与复盘（事件日志 + 结构化摘要 + 检索）
- 内容形态：支持模组/脚本的结构化（场景包），并支持规则书 KB 检索与引用
- 操作形态：提供 Web 页面（控制台）用于登录与管理（脚本/角色卡/跑团/复盘），降低使用门槛

### 1.3 非目标（Out of Scope）

- 不做通用助理：跑团以外的任务（写论文/查旅游/做报表等）属于越界内容，应被门禁拒绝
- 不追求 v1 即覆盖 CoC 规则的全部细枝末节；以“可玩 + 可审计 + 可扩展”为优先
- 不要求玩家学习大量命令：关键机制变更必须显式呈现且可审计（由 KP 输出的机制/状态变化块落地；必要时要求自然语言二次确认），避免模型“脑补改状态”

## 2 v1 默认选择（决策记录）

- 规则系统：CoC 7e
- 目标：功能越多越好，但避免过度细化拖慢节奏
- 记忆：全文追溯 + 结构化摘要 + 检索（混合方案）
- 交互：自然语言优先（TRPG-only 门禁）；机制变更必须“显式可见 + 可追溯”（不要求玩家输入细粒度命令）

## 3 术语与角色

- KP：主持人（Keeper/GM），负责裁定与推动场景
- Player：玩家，发出角色行动意图
- PC：玩家角色
- NPC：非玩家角色
- Session：一次跑团会话（可跨天/跨周继续）
- Scene：场景（模组的一级叙事单元）
- 模组/场景包：可被系统读取与引用的结构化脚本
- Leads：当前可见的推进方向/线索挂钩（2–6 条为宜）
- KB：知识库（规则书/模组等资料检索）
- Ruling：本桌裁定（固化的判例，避免同题不同判）
- Account：用户账号（用于登录与资源归属）
- Campaign：跑团/团（长期组织单元，包含成员、角色卡、脚本与多个 Session）

## 4 需求（可验收）

### 4.1 FR-01 会话管理（Session）

- 支持创建/结束一个 session；支持恢复继续跑（同一团跨天/跨周不断档）
- 允许多人加入同一 session，并能区分玩家身份与角色

### 4.2 FR-02 受控自然语言交互（TRPG-only）

- 允许玩家与 KP 用自然语言进行跑团；系统负责将关键机制结果落为“可审计的事件”（roll/damage/san/luck/状态变化等）
- 关键机制变更必须显式呈现（输出结构化的机制/状态变化块），但不要求玩家输入细粒度命令

### 4.3 FR-03 核心机制（检定/战斗/追逐）

- 战斗：伤害、重大伤害、濒死（简化轨道）、急救/治疗的基本流程
- 追逐与复杂枪械：按“场景触发”启用（不是全程强制复杂化）

### 4.4 FR-04 角色卡与状态字段（永久/临时）

- 永久字段：属性/技能/装备/背景/关系等
- 临时字段：HP/SAN/Luck、伤口/疯狂/异常状态、当场获得线索、短期目标

### 4.5 FR-05 模组/脚本上传与结构化

### 4.6 FR-06 长团记忆（长时间后仍知道以前聊过什么）

- 全文事件日志：保存每句对话、每次掷骰与裁定（可追溯）
- 摘要体系：检查点摘要 + 场景摘要 + session 总结（结构化，便于长期重建）

### 4.7 FR-07 线索与 Leads 机制（防跑偏但不强制铁路）

### 4.8 FR-08 多人桌面流程（并发与可见性）

- 聚光灯/队列规则：谁先行动、插话/打断如何处理、并发动作如何排序

### 4.9 NFR-01 可审计与可复盘

- 掷骰、SAN、伤害等关键结果必须可追溯（何时、因何、如何计算）
- 允许回放关键节点（至少能“恢复到某个场景结束时”）

### 4.10 NFR-02 TRPG-only 门禁与越界拒绝

- 在自然语言交互下，系统不应被引导去做非跑团任务
- 任何越界内容：统一拒绝模板 + 指令引导

### 4.11 FR-09 Web 页面（控制台）

- 提供 Web UI 作为 CLI 的替代入口：用户可以在页面中进行登录、管理资源、进行跑团交互
- Web UI 覆盖最小可玩闭环：输入（自然语言）→ 输出（含 `[State]/[Next]`）→ 状态/复盘查看

### 4.12 FR-10 用户登录与权限（Auth & RBAC）

- 支持登录/登出；未登录用户不能访问管理类页面
- 资源归属必须清晰：脚本/角色卡/跑团等资源必须绑定 Account 或 Campaign
- 最小权限模型：Campaign 内至少区分 KP（管理员）与 Player（成员），并执行鉴权与视角过滤

### 4.13 FR-11 脚本管理（Script Library）

- 支持脚本/场景包的上传、列表、查看、更新元信息、删除/归档
- 上传后必须进行场景包校验；校验失败的脚本不能用于开团
- 支持版本（最小可用：同一脚本多版本可选；Campaign 绑定到某一版本）

### 4.14 FR-12 角色卡管理（Character Sheets）

- 支持角色卡创建/编辑/归档/导入导出（最小：JSON）
- 字段必须遵循 M0-C 冻结的最小字段集；关键数值变化必须可审计
- 支持角色卡在 Campaign 内分配给玩家（controlled_character）

### 4.15 FR-13 跑团管理（Campaign Management）

- 支持创建/归档一个 Campaign，管理成员、绑定脚本、绑定/分配角色卡
- Campaign 作为长期组织单元，包含多个 Session 的启动/继续/结束与复盘

## 5 里程碑（Roadmap）

### M0：冻结规范（命令集 / 场景包 / 状态字段）

- 验收：命令与越界拒绝规则确定；场景包字段固定；状态字段固定

### M1：单人单场景可跑（最小闭环）

- 覆盖：FR-01、FR-02（基础拒绝与机制显式呈现）、FR-03（基础检定+奖励惩罚）、FR-04（基础角色卡）
- 验收：玩家能行动/检定；KP 能推进一个场景并更新状态

### M2：多人桌面流程（并发与可见性）

- 覆盖：FR-01（多人）、FR-08（聚光灯/队列）、FR-04（权限）、FR-02（机制变更必须可审计）
- 验收：2–4 名玩家同团；KP-only 不泄露；并发行动有明确排序

### M3：长团记忆（可检索 + 可复盘 + 不泄露）

- 覆盖：FR-06（混合记忆：全文事件日志 + 结构化摘要 + 检索）、NFR-01（可审计/可复盘）、FR-07（线索账本可检索）
- 验收：中断很久后恢复，KP 与玩家都能“按各自视角”检索回忆关键人名/线索/承诺；KP-only 不泄露

### M4：模组/规则书知识库（可读模组、可查规则）

- 覆盖：FR-05（场景包上传与引用）、FR-03（规则查询支撑）、FR-06（将模组/规则书纳入检索）
- 验收：规则解释可引用规则书片段；剧情推进能稳定引用模组要素

### M5：CoC 全功能补齐（v1“越多越好”落地）

- 覆盖：FR-03 扩展（推骰、花幸运、SAN/疯狂细化、战斗完整流程、追逐场景触发、简化枪械细则）
- 验收：跑一个含调查+战斗+SAN 冲击（可选追逐）的短模组，规则运转完整且可审计

### M6：体验打磨（防卡死与节奏控制）

- 覆盖：FR-07（线索账本+leads）、更好的拒绝与引导（NFR-02）
- 验收：玩家发散探索不会长期无事可做；KP 能稳定给出 2–4 个推进方向且不强推铁路

### M7：Web 控制台（登录 / 管理 / 跑团操作）

- 覆盖：FR-09（Web 页面）、FR-10（登录与权限）、FR-11（脚本管理）、FR-12（角色卡管理）、FR-13（跑团管理）
- 验收：用户可在 Web 登录；可创建 Campaign 并管理脚本/角色卡；可启动与继续 Session，并在页面中进行自然语言交互与查看状态/复盘

## 6 详细规格（Milestone Specs）

> 说明：除 M0 中冻结的“最小用户命令清单”外，本章后续出现的 `/xxx` 多数为内部动作/API/事件类型的占位符，用于实现与审计；不强制要求玩家输入这些细粒度命令，玩家默认以自然语言推进。

### 6.1 M0：冻结规范（详细）

#### M0-A：最小用户命令 + 语义交互（TRPG-only 门禁）

- A1 输入允许自然语言；系统默认将其解释为“跑团内发言/行动/提问”，不强制命令前缀
- A2 支持玩家身份与扮演对象切换：同一 session 可区分“玩家名”和“角色名”
- A3 区分 IC 与 OOC：允许自然语言标记（例如 `OOC:`）或命令快捷键；避免把桌外沟通误判为角色行动
- A4 任何机制变更必须“显式可见 + 可追溯”：掷骰、伤害、SAN/Luck 变动、物品增减、角色卡字段修改都必须落为机制事件并在输出中汇总
- A5 KP-only 指令与玩家指令分离：玩家不能直接调用 KP-only（例如强制改场景、读取秘密线索）
- A6 越界拒绝统一：非跑团诉求统一拒绝模板 + 引导到可用跑团命令

> 兼容性约定：后续规格中出现的 `/roll`、`/damage`、`/combat start` 等写法，视为“内部动作/API/事件类型”的占位符，不强制要求玩家直接输入；v1 对玩家暴露的命令应尽量少。KP 不需要使用命令。

##### 推荐的最小用户命令清单（M0 必须冻结）

- 会话：`/create_session`、`/resume_session <id>`、`/end_session`
- 入团：`/join <player>`、`/as <player>`（声明“当前说话人/操控者”）
- 行动：`/act "<行动/意图>"`（可选；自然语言默认等价）
- 规则：`/rule "<query>"`（可选；自然语言提问也可触发检索）
- 沟通：`/say "<text>"`（可选公开发言）、`/whisper <player> "<text>"`（可选私聊）
- 状态查询：`/status`（HP/SAN/Luck/异常/当前位置/当前 leads）
- 复盘：`/recap`（最近摘要/当前场景摘要/当前线索账本）
- 场景：`/scene next`、`/scene set <scene_id>`（KP-only）
- 安全：`/safety`、`/pause`、`/resume`、`/retcon "<修正内容>"`（KP确认后生效）
- 求助/退出：`/help`、`/quit`

#### M0-B：模组“场景包”格式（上传脚本的冻结字段）

- B1 模组必须有元信息：标题、年代/地点、推荐玩家数、风格（调查/战斗/生存）、KP-only 标签
- B2 场景是一级单元：每个场景必须有 `scene_id`、入口触发、退出条件
- B3 每个场景必须提供 2–4 条“可移动线索”（玩家走偏也能拿到线索，不会卡死）
- B4 NPC 必须结构化：姓名/外观/动机/秘密（KP-only）/与线索的关联
- B5 地点必须结构化：可互动点、可发现的线索、危险/代价
- B6 失败前进：关键行动失败时也要能推进，只是代价不同（时间、SAN、资源、敌对度）
- B7 计时器/升级：至少一个“压力源”随时间推进（怪异加深、警方介入、证据消失等）
- B8 手递物/文本证据可引用：线索可以挂载 handout 文本/图片链接（先不做图片也要留字段）

#### M0-C：状态字段清单（角色卡 + 世界状态 + 线索账本）

- C1 角色卡（PC）最小字段
  - 基础：姓名/职业/年龄/居住地/背景要点
  - 属性：STR CON DEX APP POW INT SIZ EDU
  - 派生：HP、MP、SAN、Luck、Move
  - 技能：技能名→数值（支持自定义技能）
  - 状态：伤口/重大伤害标记/疯狂状态/长期创伤/临时加减值
  - 装备与现金（建议简化记录，不做负重细化）
- C2 世界/场景状态最小字段
  - 当前场景、当前地点、当前时间段（粗粒度即可）
  - 重要 NPC 列表 + 对 PC 的态度
  - 当前威胁/倒计时（计时器）
  - 当前 leads（2–6 条，必须对玩家可见）
- C3 线索账本最小字段
  - clue_id、线索文本、来源（场景/NPC/手递物）、可信度、归属（谁知道）、指向 leads、是否已验证
- C4 可见性规则冻结：公开/仅KP/仅某玩家 三档，默认“线索公开、秘密KP-only”

#### M0 验收标准（做完即可进入 M1 开跑）

- 命令集与越界拒绝规则确定（玩家知道怎么说话、系统知道怎么拒绝）
- 场景包字段固定（可以开始写/转换模组）
- 状态字段固定（可以开始建卡、记录线索、做摘要）

### 6.2 M1：单人 + 单场景 可玩循环（Playable Loop in One Scene）

> 本章目标：让 1 名玩家在 1 个场景内“能玩、能结局、可审计”。交互默认自然语言；命令只是可选快捷键。KP 不需要“使用命令”，KP 只需要把机制过程与状态变化以稳定结构输出并写入事件日志。

#### M1-A：单人场景循环（核心叙事与门禁）

##### 目标

- 让玩家用自然语言推进叙事；让系统用“结构化输出 + 机制事件”推进状态。
- 保证 TRPG-only：非跑团请求一律拒绝，并给出可行的跑团内下一步命令。

##### 用户命令（可选快捷键，越少越好）

- `/act "..."`：可选；自然语言默认等价。
- `/help`：列出命令与示例。
- `/status`：输出当前场景摘要（目标/旗标/关键实体/开放 leads）。
- `/history [n]`：输出最近 n 条“可见对话 + 机制事件摘要”。
- `/recap`：输出更长的阶段性摘要（场景/线索/目标）。
- `/quit`：结束会话并输出会话总结。

##### 需求（Requirements）

- 输出结构稳定（便于验收与自动测试）：每轮至少包含三个固定块：
  - `[KP]`：叙事与裁定（必须）
  - `[State]`：本轮状态变化（若无变化也要显式说明 “no change”）
  - `[Next]`：下一步引导（必须给问题或 2–4 个可选行动建议）
- 门禁与一致性：
  - 所有“会改变数值/条目/战斗态/追逐态”的变更不得静默发生；必须在输出的机制/状态变化块中显式呈现并写入事件日志（必要时要求自然语言确认）。
  - 场景内实体一致性：NPC/道具/线索不能凭空出现；若需要引入，必须先在 `[State]` 中以明确新增条目落地。
  - 若模块提供 `constraints`，玩家输入违反时必须拒绝并给替代方案（不推进状态）。
- 结束条件（单场景内必须可达）：
  - 玩家主动 `/quit`
  - 成功结局：至少一种输入序列能达成 goals 或 `scene_progress` 阈值
  - 失败结局：至少一种输入序列能触发不可逆后果并结束（同样需记录原因）

##### 验收标准（Acceptance Criteria）

- 用仅 1 个场景的模块启动：能完成 ≥5 轮自然语言行动循环（可混用 `/act`），不崩溃、不丢状态。
- `/help`、`/status`、`/history 3`、`/quit` 均可用且不破坏状态。
- 给出一条明确越界输入（跑团外请求）：系统必须拒绝，输出包含 `[Refusal]`（或在 `[KP]` 中用稳定格式标识拒绝），且状态不变。
- `/quit` 输出会话总结至少包含：场景名、主要事件 3 条、最终 goals/flags、关键数值（HP/SAN/Luck 若存在）。

#### M1-B：检定闭环（语义触发，事件可审计）

##### 目标

- 在单场景内跑通：宣言 → 要求检定 → 掷骰 → 结果 →（可选）推骰/花幸运 → 后果 → 状态落地。
- 所有关键数值变化（SAN/Luck/HP）与检定事件可追溯：能指向“哪次检定/哪次伤害”。

##### 用户侧交互（不要求命令）

- 玩家用自然语言描述“做什么/用什么技能或属性/是否愿意冒险或推骰/是否花幸运”。
- KP 在输出中展示检定与结算，并将结果写入机制事件日志与 `[State]`。

##### 内部机制事件（用于审计；不要求玩家输入）

- `roll`：skill/stat、难度、奖惩骰、目标值、骰面/展开、成功等级、大成功/大失败
- `push_roll`：与上一条 roll 的关联、风险承诺文本、推骰结果与“更坏后果”
- `luck_spend`：花费点数、用途、前后剩余、对结果的影响
- `san_check` / `san_loss`：reason、loss 表达式与最终扣减
- `damage` / `hp_change`：来源、表达式、最终数值、目标

##### 需求（Requirements）

- 检定请求必须“可选择”：KP 要求检定时必须说明：检什么、成功会怎样、失败会怎样；若玩家未明确技能/属性，KP 必须给出 1–3 个可选项并解释差异。
- `/roll` 输出必须包含稳定字段：目标值、掷骰明细（含奖/惩骰展开或简述）、成功等级、是否大成功/大失败（如适用）。
- 奖励/惩罚骰互斥：同一条 `/roll` 不允许同时 `bonus` 与 `penalty`；出现时拒绝并提示二选一。
- 推骰闭环：
  - 只有被标记为“可推”的检定允许 `/push`；不可推项可先以黑名单实现。
  - 推骰失败必须触发“更坏后果”（不是原地失败），且写入 flags/leads/伤害/SAN 中至少一项。
- 幸运闭环：
  - 花幸运必须引用“最近一次检定或伤害事件”，并记录：花了多少、剩余多少、用途。
  - 花幸运后必须在 `[State]` 中体现并解释“结果被改成什么”。
- SAN 与伤害：
  - `/san`、`/damage` 必须携带可追溯 reason/source；落地到可审计的历史事件。

##### 验收标准（Acceptance Criteria）

- 成功路径：至少覆盖 1 次奖励骰与 1 次花幸运，并产生 1 条新线索（写入线索账本或 flags，且可在 `/status` 中看到）。
- 失败变更坏路径：至少覆盖 1 次推骰失败，并触发 `/damage` 或 `/san`（建议两者都出现）。
- 门禁：自然语言尝试“直接改幸运/直接扣 SAN/直接扣 HP”不得直接落地；必须由 KP 以机制事件显式结算并记录（必要时要求确认）。

#### M1-C：战斗闭环（进入战斗 → 回合 → 对抗 → 结算 → 结束）

##### 目标

- 单场景内跑通战斗的最小可玩回合制流程，并保证每一步可审计。

##### 命令

- （无强制用户命令）玩家用自然语言声明行动；KP 自动管理战斗态/回合/对抗与结算，并以机制事件输出。

##### 需求（Requirements）

- 战斗态是显式状态：KP 必须在输出中明确“已进入战斗/已结束战斗”，并在状态中维护战斗态标记。
- 回合与当前行动者清晰可见：每轮输出必须标出“当前行动者/下一位”。
- 近战对抗最小规则：攻击与防御双方都需检定，按成功等级对比裁定；输出必须解释对抗结论。
- 伤害落地必须可审计：命中后必须显式给出伤害结算过程，并在 `[State]` 中落地 HP 变化（并写入机制事件日志）。
- 重伤/濒死（可简化但要有状态位）：触发时必须在 `[State]` 写入标记（例如 `major_wound=true`、`dying=true`），并在叙事中体现行动限制。
- 治疗同样可审计：必须引用受伤事件并完成一次治疗结算（含检定与后果），成功/失败都要落地后果并写入机制事件。

##### 验收标准（Acceptance Criteria）

- 获胜路径：至少 1 次对抗结算 + 1 次伤害落地 + 1 次战斗结束小结（关键事件 3 条、PC 最终 HP）。
- 濒死路径：至少出现 1 名 PC 进入濒死/重伤阈值，并通过急救/医疗形成“稳定或失败后果”的落地记录。
- 门禁：战斗相关 HP/伤害/治疗不得静默改；必须显式呈现并落地到 `[State]`。

#### M1-D：追逐闭环（追逐态 + 距离/压力 + 失败前进）

##### 目标

- 单场景内跑通追逐：初始化 → 回合推进 → 障碍/代价 → 结束（逃脱/抓获/失败前进进入新局面）。

##### 命令

- （无强制用户命令）玩家用自然语言描述“追/逃/抄近路/制造障碍”等；KP 自动维护追逐态与距离/压力变量，并以机制事件输出。

##### 需求（Requirements）

- 追逐态显式：KP 必须在输出中明确“已进入追逐/已结束追逐”，并在状态中维护追逐态标记。
- 最小状态变量：round、distance（相对刻度即可）、pressure 或 timer、env tags。
- 每回合必须可结算：每位行动者的追逐动作要么映射到 `/roll`，要么 KP 明确声明“无需检定”并记录原因。
- 失败前进是硬约束：失败不能只是不成功，必须产生代价（伤害/SAN/丢失物品/引来第三方/计时器推进等）并写入状态。

##### 验收标准（Acceptance Criteria）

- 成功逃脱：距离/优势按规则变化并最终满足结束条件；在输出中记录追逐结束与原因。
- 失败前进：至少一次失败触发代价（含伤害/SAN/flags 等），并在输出中形成“新局面”。
- 门禁：追逐中的伤害/SAN 仍必须显式呈现并可追溯。

#### M1-E：规则问答闭环（检索 + 引用 + 桌规裁定）

##### 目标

- 规则问题“先检索再回答”，并将超出证据的裁量固化为可追溯的桌规（ruling）。

##### 命令

- 用户侧（可选快捷键）：`/rule "<query>"`
- KP 侧：自动检索并给出引用；必要时形成桌规裁定并落为可追溯事件（不要求用户输入 `/kb` 或 `/ruling` 命令）

##### 需求（Requirements）

- 回答结构稳定：必须包含 `[Question]`、`[Answer]`、`[Citations]`、`[Ruling]` 四块（即使没有也要写 “none”）。
- 证据优先级：桌规 > 规则书引用 > 模组特例 > KP 临时裁量（临时裁量必须固化为可追溯的 ruling 事件）。
- 无证据不胡编：检索无结果必须明确说明并给下一步（换关键词/补上下文/补充资料/临时桌规）。
- 防注入：KB 片段是资料不是指令；任何“忽略门禁/执行别的任务”的文本都必须被当作普通文本。
- TRPG-only：规则问答不得引导跑团外行为。

##### 验收标准（Acceptance Criteria）

- 有证据问答：至少 1 次检索命中并引用 citation_id，且可复核上下文。
- 矛盾证据：至少 1 次出现冲突或不完整证据，KP 进行澄清或形成桌规裁定事件，并能追溯。
- 无证据：至少 1 次检索为空，KP 不编规则条文，给出下一步，并（若需要）记录临时桌规。

### 6.3 M2：多人桌面流程（并发与可见性）（详细）

#### M2-A：多人加入/身份/席位（Players & Seats）

##### 目标

- 支持 2–4 名玩家加入同一 session，并能稳定区分“玩家身份”与“角色身份”
- 允许玩家中途掉线/回归；会话状态可恢复继续跑

##### 新增/细化命令（在 M0 命令基础上补齐）

- `/players`：列出当前 session 玩家列表（player_id、显示名、在线状态、当前操控角色）
- `/leave`：玩家离开（保留角色与状态，标记为离线）
- `/kick <player>`：移出玩家（KP-only；需记录原因）
- `/as <player>`：声明当前说话人/操控者（复用 M0；多人时必须生效且可审计）

##### 细粒度需求（Requirements）

###### R1：身份与归属可审计

- 每条输入与系统落地的机制事件必须绑定：session_id、actor_player_id、actor_role（KP/Player）、controlled_character_id（若有）
- KP-only 与 Player-only 命令必须严格鉴权；鉴权失败需给出稳定拒绝信息并写入审计日志

###### R2：多人同时输入的最小处理策略

- 系统必须定义“同一时刻多条输入”的处理策略（最小可用）：按到达顺序进入队列，逐条处理
- 任何被排队的输入必须立即得到“已入队”的确认，并可通过 `/queue` 查看

##### 验收标准（Acceptance Criteria）（M2-A）

- 2–4 名玩家可用 `/join` 加入并用 `/players` 查看到一致结果
- 任意玩家 `/leave` 后不会丢失角色卡与线索归属；再次 `/resume_session <id>` 后能继续
- 玩家尝试执行 KP-only（如 `/kick`）会被拒绝，且拒绝可被审计复盘

#### M2-B：聚光灯/队列（Spotlight & Turn Queue）

##### 目标

- 在多人桌面中给出明确的“谁在行动”的规则，减少插话与并发造成的状态冲突
- 让 KP 能控制节奏：推进当前行动者、允许插话、或暂缓某人行动

##### 新增/细化命令

- `/spotlight show`：显示当前聚光灯（当前行动者、等待队列、允许插话状态）
- `/spotlight next`：推进到下一行动者（KP-only；或由规则自动）
- `/queue`：查看输入/行动队列（按时间顺序）
- `/queue drop <id>`：丢弃某条排队输入（KP-only；需记录原因）
- `/interrupt request`：申请插话（玩家命令；仅登记，不直接改变状态）
- `/interrupt allow <player>`：允许某玩家插话一次（KP-only）

##### 细粒度需求（Requirements）

###### R1：聚光灯是“状态变量”

- 状态中至少包含：
  - `spotlight.current_player`
  - `spotlight.queue`（按行动者顺序，而不是按消息顺序）
  - `spotlight.interrupt_requests`（待处理插话请求）
- KP 每轮输出应包含一个稳定块（例如 `[Spotlight]`）提示当前行动者与下一步

###### R2：并发写冲突的最小避免

- 当非当前聚光灯玩家直接提交会改变状态的命令（如 `/roll`、`/sheet set`、`/inv add` 等）时：
  - 系统必须拒绝或入队（两者二选一，但行为需一致）
  - 必须提示“等待聚光灯”或“已排队”，并给出 `/spotlight show` 与 `/queue` 的指引
- KP 可用 `/interrupt allow` 临时允许一次“越过聚光灯”的行动，但该授权必须写入审计

###### R3：插话/打断的最小规则

- 插话不应直接改变世界状态；插话只允许：补充信息、短句提问、对当前行动的协助声明
- 若插话涉及机制变更（如“我也要开枪”），必须排队到其回合或由 KP 显式授权

##### 验收标准（Acceptance Criteria）（M2-B）

- 预置 3 名玩家同时发送输入：系统能稳定排队并按规则处理，不产生“同一条状态被互相覆盖”
- 非聚光灯玩家提交 `/roll`：系统能稳定拒绝或入队，并给出清晰提示
- KP 可用 `/spotlight next` 推进桌面流程；可用 `/queue drop` 丢弃一条输入且留痕

#### M2-C：可见性（KP-only / Player-only）与信息泄露防护

##### 目标

- 确保 KP-only 内容不会泄露给玩家；同时支持“仅某玩家可见”的线索/私密信息
- 让多人状态查询（`/status`、`/sheet show`、`/recap`）遵循可见性规则

##### 新增/细化命令

- `/whisper <player> "<text>"`：向某玩家发送私密信息（KP-only；写入日志但按权限隐藏）
- `/tell kp "<text>"`：玩家向 KP 私密汇报（写入日志但对其他玩家不可见）
- `/visibility set <object_ref> <public|kp|player:<id>>`：调整某对象可见性（KP-only；object_ref 可为 clue_id / handout_id / note_id）

##### 细粒度需求（Requirements）

###### R1：所有可输出内容必须经过权限过滤

- 系统在生成任何面向玩家的输出时，必须以“当前视角（viewer）”过滤：
  - KP 视角：可见全部（除安全工具/隐私策略另有限制）
  - 玩家视角：只能看到 public + 自己可见（player:<id>）的内容
- 过滤不仅作用于“最终输出文本”，也作用于：/history、/recap、/status、/sheet show、线索账本

###### R2：线索账本可见性与归属

- M0 已冻结“公开/仅KP/仅某玩家”三档；M2 必须保证该规则在多人环境真实生效
- 当线索归属为“仅某玩家”时：
  - 其他玩家的 `/recap` 不应出现该线索文本
  - KP 的 `/recap` 必须能看到该线索及其归属

###### R3：最小防泄露验收点

- KP-only 内容（例如 NPC 秘密、隐藏线索、未来场景钩子）不得通过任何“摘要/复盘/历史”泄露到玩家输出
- 若玩家试图通过提示注入诱导输出 KP-only 内容，系统必须拒绝并引导回跑团命令

##### 验收标准（Acceptance Criteria）（M2-C）

- KP 用 `/whisper <player>` 发送秘密线索：
  - 目标玩家能看到
  - 其他玩家看不到（包括 `/history` 与 `/recap`）
  - KP 可在审计视角复盘到该事件
- 玩家用 `/tell kp` 汇报：仅 KP 可见，其他玩家不可见

#### M2-D：多人状态查询与复盘（Per-Viewer Recap）

##### 目标

- 在多人会话中，保证 `/status`、`/recap` 的输出“按视角一致”且不泄露
- 支持长 Session 中断后恢复：每个玩家都能恢复到正确的“我知道什么/我能做什么”

##### 新增/细化命令

- `/status [me|all]`：默认 `me`（只看与自己相关/可见的摘要）；KP 可用 `all` 查看全局
- `/recap [me|public|all]`：默认 `me`；KP 可用 `all`

##### 细粒度需求（Requirements）

###### R1：同一事件，多种视角摘要

- 同一个“事件日志”应能派生出：KP 视角摘要、玩家视角摘要（每玩家不同）
- 当视角不同导致信息缺失时，系统应保持叙事连贯（允许用“你隐约感觉…”等模糊表述，但不得泄露具体 KP-only 内容）

###### R2：恢复一致性

- `/resume_session <id>` 后：
  - KP 能恢复全局状态（场景、计时器、全线索账本、所有 flags）
  - 玩家只能恢复到其可见子集（含自己私密线索）

##### 验收标准（Acceptance Criteria）（M2-D）

- 预置：同一 session 中同时存在 public clue 与 player-only clue：
  - 不同玩家执行 `/recap` 得到不同内容（符合可见性）
  - KP 执行 `/recap all` 能看到完整内容
- 中断后恢复：任意玩家恢复后仍能看到“自己私密线索 + public 线索”，且不会突然看到别人的私密线索

#### M2-E：共享状态写入与所有权（Sheets / Inventory / Clues / Flags）

##### 目标

- 定义多人环境下“谁能改什么”的最小规则，避免角色卡/物品/线索被他人误改
- 对可能冲突的写入（几乎同时修改同一字段）提供可解释的处理方式

##### 新增/细化命令

- `/sheet show [char]`：默认查看当前操控角色；KP 可指定 `char`
- `/sheet set [char] <field>=<value>`：默认修改当前操控角色；指定 `char` 仅允许 KP
- `/inv add [char] "<item>"`、`/inv remove [char] "<item>"`：同上（指定 `char` 仅允许 KP）

##### 细粒度需求（Requirements）

###### R1：写入权限（Owner / KP）

- 角色卡与物品的“直接写入”遵循最小权限：
  - 玩家：只能修改自己当前操控角色的角色卡/物品（且仍需显式可审计的变更记录；必要时要求确认）
  - KP：可以修改任意角色（需可审计：修改了谁、改了什么、为什么）
- 线索账本：
  - 玩家只能创建/补充“自己视角的记事”（如果允许），但不能把 KP-only 线索改成 public
  - 可见性与归属的修改（`/visibility set`）必须是 KP-only

###### R2：冲突写入的最小处理

- 对同一对象/字段的写入必须可检测冲突（最小方案二选一即可）：
  - 严格串行：所有写入进入同一队列，按顺序落地（推荐 v1）
  - 乐观并发：基于 `revision` 或时间戳检测冲突，冲突时拒绝并提示重试
- 冲突发生时必须提示：冲突对象、冲突字段、当前值、建议动作（例如先 `/sheet show` 再重试）

###### R3：机制事件落地必须绑定目标

- `/roll`、`/damage`、`/san`、`/luck` 等机制事件在多人环境下必须能明确绑定到“哪个角色”
- 默认绑定到当前操控角色；若 KP 需要对 NPC/其他角色落地，应通过 KP-only 方式显式指定

##### 验收标准（Acceptance Criteria）（M2-E）

- 玩家 A 尝试 `/sheet set` 修改玩家 B 的角色：被拒绝并给出原因
- KP 可指定 `char` 修改任意角色：修改被记录并能在复盘中追溯
- 两名玩家几乎同时修改同一字段：系统不会静默覆盖；要么串行落地可解释，要么检测冲突并拒绝其一

#### M2-F：多人战斗/追逐的回合与桌面流程联动

##### 目标

- 让多人战斗/追逐时的“回合顺序”和“聚光灯”一致，避免同时多个人宣告攻击导致混乱
- 确保战斗/追逐中的机制变更（攻击、伤害、治疗、追逐行动）仍遵守门禁与可审计

##### 新增/细化命令

- `/turn show`：显示当前回合结构（当前行动者、下一位、回合数；战斗/追逐态均可用）
- `/turn next`：在战斗/追逐态下推进行动者（KP-only；可与现有 `/turn next` 复用）

##### 细粒度需求（Requirements）

###### R1：进入战斗/追逐时切换“回合驱动”

- 当进入 `/combat start` 或 `/chase start` 后：
  - 聚光灯应切换为“按回合行动者”为准（可以是玩家，也可以是 NPC）
  - 非当前行动者提交会改变状态的命令（攻击/治疗/追逐行动）必须拒绝或入队

###### R2：宣告与结算的最小一致性

- 当前行动者可以用 `/act` 宣告行动意图；实际机制落地仍通过对应命令链（例如 `/attack` → `/roll` → `/damage`）
- 若多个玩家同时宣告（输入顺序接近）：
  - 只有当前行动者的“可改变状态”的命令会被立即执行
  - 其他人的命令必须排队或拒绝，并提示等待 `/turn next`

##### 验收标准（Acceptance Criteria）（M2-F）

- 预置 2 名 PC + 1 名 NPC 的战斗：
  - 系统能稳定约束“只有当前行动者能攻击/治疗”
  - `/turn show` 与 `[Spotlight]`/回合提示一致
- 预置追逐中 2 名玩家同时提交追逐行动：系统不会并发写坏 distance/pressure 等状态变量

#### M2-G：暂停/安全工具在多人桌面的行为

##### 目标

- 明确 `/pause`、`/resume`、`/retcon` 在多人环境的语义，避免有人暂停后他人仍能推进机制
- 让“安全中断”与“回滚修正”可审计且不造成权限泄露

##### 新增/细化命令

- `/pause`：任何玩家可发起全桌暂停（写入全局状态 `paused=true`）
- `/resume`：仅 KP 可恢复（或要求全员确认；两者二选一，但必须固定规则）
- `/retcon "<text>"`：任何人可提出，但必须 KP 确认后生效（保留提案与最终版本）

##### 细粒度需求（Requirements）

###### R1：暂停态的输入处理

- 当 `paused=true` 时：
  - 禁止任何会改变规则状态的命令落地（roll/damage/san/luck/sheet/inv/combat/chase 等）
  - 允许最小集合：查看状态（`/status`、`/recap`）、求助（`/help`）、以及非机制的沟通（可用 `/act` 但必须标记为 OOC/暂停对话）

###### R2：回滚修正（Retcon）可追溯

- `/retcon` 的生效必须产生一条“修正事件”：
  - 提案人、提案内容、KP 采纳内容、影响范围（哪一段历史/哪一条事件）
  - 若会影响既有机制结果（如 HP/SAN 变化），必须要求通过命令重新落地或明确标注“人工修正”

##### 验收标准（Acceptance Criteria）（M2-G）

- 任意玩家 `/pause` 后，其他玩家尝试 `/roll`：会被拒绝并提示当前为暂停态
- KP `/resume` 后桌面可继续推进；暂停期间的队列输入不会被错误执行
- 发起一次 `/retcon` 并被 KP 采纳：复盘中能看到提案与最终修正内容

#### M2-H：多人鲁棒性（掉线、重复输入、幂等性）

##### 目标

- 让多人长 Session 在掉线/重复发送/重连的情况下仍能保持状态一致
- 避免“同一条命令被重复执行两次”导致的数值漂移

##### 细粒度需求（Requirements）

###### R1：输入去重（幂等性）

- 系统应能识别并去重短时间内的重复输入（最小方案：对同一 player 的相同命令文本在时间窗内去重，并提示已处理）
- 对高风险命令（`/damage`、`/san`、`/luck`、`/sheet set` 等）重复执行必须有保护：禁止静默重复扣数

###### R2：掉线后的队列处理

- 玩家离线时，其未执行的“会改变状态”的排队命令应被自动暂停或丢弃（两者二选一，但规则必须固定并可解释）
- 玩家重连后，可通过 `/queue` 查看是否仍有待处理输入，并可选择重发

##### 验收标准（Acceptance Criteria）（M2-H）

- 同一玩家因网络问题重复发送 `/luck 5`：只会被落地一次，另一条会被提示为重复
- 玩家离线后仍在队列中的攻击命令不会在其离线期间被执行

#### M2 验收标准（做完即可进入后续里程碑）

- 2–4 名玩家同团可稳定跑 10+ 轮：
  - 聚光灯/队列规则明确且可操作
  - 并发输入不会导致状态冲突或泄露
- KP-only 与 player-only 权限边界稳定：
  - KP-only 内容不泄露（含 history/recap/status）
  - 仅某玩家可见信息在多人复盘中正确隔离
- 可审计与可复盘（NFR-01）：
  - 任意一次敏感输出可追溯到产生原因与权限过滤结果（至少能解释“为何 A 看到了、B 没看到”）
- 多人写入与机制幂等：
  - 角色卡/物品/关键数值不会被重复执行或被他人误改
  - 战斗/追逐状态在多人并发输入下仍能保持一致

### 6.4 M3：长团记忆（可检索 + 可复盘 + 不泄露）（详细）

#### M3-A：事件日志（Event Log）与索引（Append-only）

##### 目标

- 将 session 中的所有关键事件以“可追溯、可检索、可审计”的形式落盘：对话、命令、掷骰、裁定、状态变更
- 明确日志的最小结构，保证后续摘要/检索/复盘都有稳定数据源

##### 新增/细化命令

- `/history [n]`：查看最近 n 条（按当前视角过滤）
- `/log show <event_id>`：查看单条事件详情（KP-only 或按权限过滤后的版本）

##### 细粒度需求（Requirements）

###### R1：事件最小字段（冻结）

- 每条事件至少包含：
  - `event_id`、`timestamp`
  - `session_id`、`scene_id`（若有）
  - `actor_player_id`、`actor_role`（KP/Player）、`controlled_character_id`（若有）
  - `type`（chat/command/roll/ruling/state_change/recap/checkpoint 等）
  - `payload`（结构化；原始文本也需保留）
  - `visibility`（public/kp/player:<id>）
- 事件日志必须是 append-only：允许通过“修正事件（retcon）”覆盖语义，但不允许静默篡改历史

###### R2：状态变更必须可追溯到事件

- 任何持久化状态变更（HP/SAN/Luck/flags/线索/角色卡/物品/场景切换）必须能指回产生它的 event_id
- 若存在“人工修正”，必须以明确 type 记录原因与影响范围

##### 验收标准（Acceptance Criteria）（M3-A）

- 任意一次 `/damage`、`/san`、`/luck` 的落地，都能通过 `/log show` 查到：原因、计算过程（如有）、前后数值、event_id
- 进行一次 `/retcon`：日志中能看到“修正事件”，且原事件不会消失

#### M3-B：结构化摘要（Checkpoints / Scene / Session）

##### 目标

- 为长 Session 提供可重建的“低成本记忆”：检查点摘要、场景摘要、session 总结
- 摘要必须结构化，便于检索与“按视角过滤”

##### 新增/细化命令

- `/checkpoint`：生成一个检查点（KP-only；写入结构化摘要）
- `/recap [me|public|all]`：复用 M2-D；M3 要求输出稳定结构（摘要 + leads + 未解决悬念）

##### 细粒度需求（Requirements）

###### R1：摘要最小字段

- 每个检查点摘要至少包含：
  - `checkpoint_id`、关联 `event_id` 范围（from/to）
  - `scene_id`、时间线（粗粒度即可）
  - 主要事件 3–7 条（bullet）
  - 当前 leads（2–6 条）
  - 关键承诺/未竟事项（promises/todos）
  - 关键数值快照（HP/SAN/Luck 及重要状态标记）

###### R2：摘要频率与触发

- 最小触发策略（二选一即可，但必须固定）：
  - KP 显式 `/checkpoint`
  - 系统在“场景结束/战斗结束/追逐结束”后建议 KP 创建检查点

##### 验收标准（Acceptance Criteria）（M3-B）

- 跑 10+ 轮后生成 `/checkpoint`：检查点中能看到 leads、关键事件、关键数值快照
- 中断恢复后执行 `/recap`：能基于最近检查点与事件日志输出一致的摘要（不凭空编）

#### M3-C：记忆检索（Memory Search）与引用

##### 目标

- 支持从“事件日志 + 摘要 + 线索账本”中检索信息，并能将答案绑定到可引用的证据
- 玩家与 KP 的检索结果必须遵循可见性（不泄露）

##### 新增/细化命令

- `/memory "<query>" [topk=N]`：在本 session 记忆中检索（返回片段摘要 + 引用 id + 可见性标签）
- `/memory show <mem_id>`：展开某条记忆引用（按视角过滤；用于复核上下文）

##### 细粒度需求（Requirements）

###### R1：检索结果必须可引用

- `/memory` 返回的每条结果必须包含：
  - `mem_id`
  - 来源类型（event/checkpoint/clue）
  - 简短摘要（1–2 句）
  - 关联的 event_id 或 checkpoint_id（至少一个）

###### R2：按视角过滤（不泄露）

- 玩家调用 `/memory` 只能检索到其可见内容（public + player:<id>）
- KP 调用 `/memory` 可见全部；但若 KP 选择对玩家复述，必须通过叙事而不是直接贴 KP-only 原文

###### R3：最小“防幻觉”约束

- 当模型使用记忆回答问题时，必须优先引用 `/memory` 的结果；没有证据时要明确不确定并建议用关键词再检索

##### 验收标准（Acceptance Criteria）（M3-C）

- 玩家问“我们答应过谁什么？”：
  - KP 或玩家执行 `/memory "答应"` 能检索到承诺条目，并能用 `/memory show` 复核
- 存在 player-only 线索时：
  - 非该玩家执行 `/memory` 搜不到该线索
  - KP 能搜到且不会因摘要/检索而泄露给其他玩家

#### M3-D：恢复与一致性（Resume Consistency）

##### 目标

- `/resume_session <id>` 后，KP 与玩家都能恢复到“正确的当前状态 + 正确的已知信息”
- 确保复盘/检索与当前状态一致：不会出现“回忆说 HP=10 但状态里 HP=6”这种漂移

##### 细粒度需求（Requirements）

###### R1：状态快照与检查点一致

- 检查点摘要中的关键数值快照必须能与当时状态对齐（通过 event_id 链可复核）
- 若后续发生 retcon/人工修正，必须：
  - 生成新的检查点或标记旧检查点已过期
  - 复盘输出必须优先使用最新有效检查点

##### 验收标准（Acceptance Criteria）（M3-D）

- 在产生一次伤害与一次 SAN 损失后创建检查点；中断后恢复：
  - `/status` 的数值与 `/recap` 的快照一致
  - `/memory` 能检索到对应事件并可引用

#### M3 验收标准（做完即可进入 M4）

- 记忆可用：
  - 10+ 轮的对话后，能用 `/recap` 与 `/memory` 找回关键人名/线索/承诺/当前 leads
- 不泄露：
  - KP-only 与 player-only 内容在 history/recap/memory 下都不会跨视角泄露
- 可审计：
  - 任意一个记忆结论都能追溯到 event_id/checkpoint_id/clue_id（至少一个）

### 6.5 M4：模组/规则书知识库（可读模组、可查规则）（详细）

#### M4-A：模组导入与场景包校验（Module Ingestion & Validation）

##### 目标

- 支持导入“模组/剧本/脚本”，并转换/落地为 M0 冻结的“场景包”结构（至少可读、可引用、可跑）
- 导入过程中必须做结构校验：缺字段可给出错误或降级策略，但不能静默吞掉关键字段

##### 新增/细化命令

- `/module upload <path|url>`：导入模组来源（先不要求实现 URL；但命令形态冻结）
- `/module list`：列出已导入模组（module_id、标题、年代/地点、推荐玩家数、版本）
- `/module show <module_id>`：查看模组元信息与场景列表（按视角过滤：KP 能看 KP-only 标签）
- `/module validate <module_id>`：运行结构校验并返回问题清单（KP-only）
- `/scene list`：列出当前模组的 scene_id 列表与简述（按视角过滤）

##### 细粒度需求（Requirements）

###### R1：导入结果必须是“可引用对象”

- 导入后的场景包必须生成稳定引用：module_id、scene_id、npc_id、clue_id、handout_id
- 所有可引用对象必须带可见性标记（public/kp/player:<id>）

###### R2：结构校验最小集

- 至少校验以下冻结字段是否存在且合法（来自 M0-B）：
  - 模组元信息（标题、年代/地点、推荐玩家数、风格、KP-only 标签）
  - 场景（scene_id、入口触发、退出条件）
  - 每场景 2–4 条可移动线索（数量不足必须警告）
  - NPC/地点结构化字段（缺失必须警告或拒绝导入，由实现选择但需固定策略）
- 校验失败时必须给出：问题类型、位置（module_id/scene_id/字段名）、建议修复

##### 验收标准（Acceptance Criteria）（M4-A）

- 导入一个最小模组（至少 2 个场景、每场景≥2 线索）：
  - `/module list` 可见；`/module show` 能列出场景
  - `/module validate` 无致命错误（或仅有明确的可接受警告）

#### M4-B：规则书/资料入库（Chunking + Citations）

##### 目标

- 支持将规则书（或规则摘录）入库为可检索片段，并提供稳定引用（citation_id）
- 为 M1.4 的规则问答提供可复核的引用基础

##### 新增/细化命令

- `/kb ingest <source> <path|url>`：入库资料（source 例如 rulebook/module/notes；v1 允许只做 rulebook）
- `/kb sources`：列出已入库来源（source_id、名称、版本/页码信息（若有）、导入时间）

##### 细粒度需求（Requirements）

###### R1：引用最小字段

- 每个可检索片段必须包含：citation_id、source_id、定位信息（页码/章节/段落号至少其一）、正文片段、可见性
- 片段正文必须以“资料”对待：不得包含可执行指令语义（见 M4-E）

###### R2：可复核

- 对任何引用，必须能通过 `/kb show <citation_id>` 看到完整上下文（至少扩展上下各一小段）

##### 验收标准（Acceptance Criteria）（M4-B）

- 导入一份规则资料后：`/kb sources` 能列出来源；`/kb show` 能展开任意 citation_id 的上下文

#### M4-C：统一检索接口（Rules + Module + Memory）

##### 目标

- 提供统一的检索命令，让 KP 在推进剧情或裁定规则时能快速找到“依据”：规则书片段、模组元素、长记忆事件
- 检索必须按视角过滤，且结果可引用

##### 新增/细化命令

- `/search "<query>" [scope=rulebook|module|memory|all] [topk=N]`：统一检索入口（按视角过滤）
- `/search show <ref_id>`：展开引用（ref_id 可以是 citation_id/mem_id/module_ref）

##### 细粒度需求（Requirements）

###### R1：结果结构稳定

- `/search` 的每条结果必须包含：ref_id、来源类型、摘要、相关度（可粗粒度）、可见性标签

###### R2：与既有命令兼容

- 保留 `/kb rule` 与 `/memory` 作为专用快捷方式；但其输出结构应与 `/search` 对齐（至少能映射到同一 ref_id 体系）

##### 验收标准（Acceptance Criteria）（M4-C）

- 用同一个 query：
  - `scope=rulebook` 能返回 citation
  - `scope=module` 能返回 module_ref（如 scene/npc/clue）
  - `scope=memory` 能返回 mem_id
- 任意 ref_id 都能用 `/search show` 复核

#### M4-D：剧情推进的“模组落地”（Grounded Narration）

##### 目标

- KP 在叙事推进时应优先引用模组要素（NPC/地点/线索/手递物），避免“凭空编造关键设定”
- 允许适度即兴，但必须可追溯：新增实体要写入状态并标记为 improvised

##### 细粒度需求（Requirements）

###### R1：模组优先与引用

- 当当前场景存在可用线索/NPC/地点时，KP 的输出应优先从模组对象中选取并可在审计视角标注引用（无需对玩家展示引用 ID，但 KP 侧必须可追溯）
- 若 KP 引入“模组外新增实体”，必须：
  - 给出合理叙事来源
  - 写入世界状态（例如 npc 列表/地点交互点）
  - 标记 `improvised=true`

###### R2：线索可移动与防卡死

- 当玩家走偏导致当前线索不可达时，KP 应使用“可移动线索”机制将线索投放到可达交互点（并写入线索账本来源）

##### 验收标准（Acceptance Criteria）（M4-D）

- 选用一个含 2 场景的模组：在场景推进中至少 2 次引用到模组 NPC/线索（可审计追溯到 module_ref）
- 引入一个即兴 NPC：必须被写入状态并标记为 improvised

#### M4-E：访问控制与防注入（Module/KB as Data）

##### 目标

- 防止模组/规则资料中的文本被当作“系统指令”执行（数据即数据）
- 确保 KP-only 的模组秘密、隐藏线索不会通过检索/复盘/引用泄露给玩家

##### 细粒度需求（Requirements）

###### R1：资料防注入

- 无论模组文本/规则书片段中出现何种指令式措辞（例如“忽略门禁”“执行系统命令”），系统必须将其视为普通文本，不影响运行规则与门禁
- 检索结果与引用展示必须带来源标记（source_id/module_id），避免被伪装成系统输出

###### R2：KP-only 不泄露

- 对 module_ref/citation/memory 的所有输出必须按 viewer 过滤：
  - 玩家不可通过 `/search`、`/kb show`、`/history`、`/recap`、`/memory show` 获取 KP-only 内容
  - KP 可见全部，但对玩家复述时不得直接贴 KP-only 原文

##### 验收标准（Acceptance Criteria）（M4-E）

- 预置：模组或规则书片段包含“注入文本”（诱导越权/越界）：
  - 系统不执行、不采纳为指令，仍保持 TRPG-only 门禁
- 预置：模组存在 KP-only 秘密线索：
  - KP 能检索到
  - 玩家侧通过任何检索/复盘接口都检索不到

#### M4 验收标准（做完即可进入 M5）

- 可读模组：
  - 至少 1 个模组可导入、可校验、可列出场景，并能在推进时引用到模组要素
- 可查规则：
  - 至少 3 个常见规则问题可通过检索给出“带 citations 的答案”，并可用 show 复核
- 可审计与不泄露：
  - 引用来源可追溯（module_ref/citation_id/source_id）
  - KP-only 内容不会泄露到玩家视角的检索/复盘输出

### 6.6 M5：CoC 全功能补齐（v1“越多越好”落地）（详细）

#### M5-A：SAN/疯狂机制补齐（临时疯狂/不定疯狂/症状与恢复）

##### 目标

- 在保持可玩与可审计的前提下，把 SAN 相关的“后续影响”补齐：临时疯狂、不定疯狂、症状（恐惧/狂躁/幻觉等）与恢复窗口
- 让 SAN 的触发、扣除、症状落地与解除都可追溯（不会只停留在数值变化）

##### 新增/细化命令

- `/san check <reason> <loss>`：进行一次 SAN 检定并落地结果（loss 支持 `a/b` 或 `a/1dN` 或 `1dN`）
- `/san apply <points> <reason>`：直接扣除 SAN（KP-only；用于无法检定或特殊事件；必须记录原因与依据）
- `/insanity apply <temp|indef> <symptom> [duration]`：落地一次疯狂状态（KP-only；写入状态标记与叙事提示）
- `/insanity clear <id>`：解除/转化某条疯狂状态（KP-only；记录原因：治疗/时间/剧情）

##### 细粒度需求（Requirements）

###### R1：SAN 结算结构稳定

- 每次 SAN 事件输出（KP 侧审计结构）至少包含：
  - reason、loss_spec、最终 loss
  - 触发依据（可引用：citation_id/module_ref/event_id 任一）
  - 前后 SAN 数值
  - 是否触发疯狂（temp/indef/none）与对应症状 id（如触发）

###### R2：疯狂状态是“可执行的约束”而非纯文案

- 疯狂状态必须写入角色状态字段，并在后续回合中对 KP 的叙事与行动建议产生约束：
  - 至少要求在每轮 `[Status]` 或 `[Next]` 中提示“当前症状影响”（短句即可）
  - 若玩家行动与症状强冲突，KP 应提示并要求玩家改述或使用 `/retcon` 走修正流程

###### R3：恢复与治疗的最小闭环

- 解除/减轻疯狂必须通过命令落地（`/insanity clear` 或等价），并记录：谁做了什么治疗/安抚/时间跨度

##### 验收标准（Acceptance Criteria）（M5-A）

- 跑一次 SAN 检定（`/san check`）并触发临时疯狂：
  - 结果可审计（reason/loss_spec/最终 loss/前后 SAN）
  - 疯狂状态写入角色卡，并在后续 3 轮输出中持续提示其影响
- 通过治疗/时间推进解除疯狂：`/insanity clear` 可追溯并更新状态

#### M5-B：战斗完整流程补齐（远程/护甲/弹药/故障/更完整对抗）

##### 目标

- 将 M1.2 的战斗从“近战为主 + 远程简化”升级到可跑短模组的完整度：
  - 远程射击与距离/掩体 penalty
  - 护甲与伤害减免（最小可用）
  - 弹药/装填（最小可用）
  - 重大成功/大失败（如你选择启用）导致的额外效果（例如故障/武器损坏）

##### 新增/细化命令

- `/rules mode <basic|standard|extended>`：规则复杂度开关（KP-only；默认 basic；M5 允许 extended）
- `/range set <target> <band>`：设置距离带（KP-only；band 可为 close/near/far/long，或整数刻度）
- `/ammo set <weapon> <current>/<max>`：设置弹药（KP-only 或玩家对自己武器；需鉴权）
- `/reload <weapon>`：装填/换弹（需处于战斗态；落地 ammo 变化）
- `/armor set <target> <value>`：设置护甲值（KP-only；用于 NPC 或临时护甲）

##### 细粒度需求（Requirements）

###### R1：远程命中与修正

- 在 extended 模式下，远程攻击结算必须明确给出 penalty 来源（距离/掩体/移动/光照等，最小可用即可）
- penalty/bonus 的来源必须可审计（写入事件 payload）

###### R2：弹药与装填的最小一致性

- 若武器被标记为需要弹药，则每次射击必须减少 ammo；ammo 为 0 时必须拒绝射击并提示 `/reload`
- 装填必须消耗一次行动（与回合/聚光灯联动，见 M2-F）

###### R3：护甲与伤害减免

- 若目标存在 armor，则 `/damage` 落地时必须计算减免并输出：原始伤害、护甲、最终伤害
- 护甲来源必须可追溯（来自角色卡/装备/模组设定/临时设置）

###### R4：故障与武器损坏（最小可用）

- 若启用大失败：必须给出清晰后果（卡壳/走火/武器损坏/暴露位置等），并写入状态或 flags

##### 验收标准（Acceptance Criteria）（M5-B）

- 远程战斗片段（至少 3 轮）：
  - distance/cover 导致的 penalty 可解释
  - ammo 会减少，ammo=0 时不能射击
  - 护甲减免被正确计算且可审计
- 触发一次故障（或等价不利事件）：能看到后果写入状态，并影响后续行动

#### M5-C：追逐补齐（多参与者/载具要素/更丰富障碍与代价）

##### 目标

- 将 M1.3 的追逐从“单场景可跑”扩展为可覆盖短模组常见情况：
  - 多追逐者/多逃跑者
  - 载具/地形标签对 Move/距离的影响（最小可用）
  - 更稳定的障碍生成/选择与代价落地

##### 新增/细化命令

- `/chase vehicle <actor> <type>`：声明载具类型（foot/car/boat 等；先做枚举即可）
- `/chase tag add <tag>`、`/chase tag remove <tag>`：环境标签调整（KP-only）
- `/chase obstacle roll`：生成一个障碍（KP-only；可基于 tag 选择障碍表）

##### 细粒度需求（Requirements）

###### R1：多参与者一致性

- distance/advantage 至少要支持“追逐阵营间的相对距离”或“每对关键对手的距离”二选一（但必须固定一种模型）
- 新加入参与者必须有初始化距离，并写入状态（不可隐式加入）

###### R2：失败前进必须落到可检索的新局面

- 追逐失败/中断必须形成新局面，并写入：
  - 进入新场景入口、丢失物品、受伤、引来第三方、计时器推进等

##### 验收标准（Acceptance Criteria）（M5-C）

- 1 名 PC + 2 名 NPC 的追逐：能稳定跑 5+ 回合，且每回合距离变化可审计
- 触发一次“失败前进”：结局不是原地踏步，且新局面可通过 `/recap` 或 `/history` 找回

#### M5-D：成长与战后流程（技能成长/医疗恢复/战利品与资源）

##### 目标

- 支持短模组结束后的最小成长闭环：技能成长（经验检定）、战后治疗与状态清算
- 保持门禁：所有角色卡修改必须通过命令，且可审计

##### 新增/细化命令

- `/xp mark <skill>`：标记一次技能成长机会（KP-only 或玩家对自己角色，按你选择的规则）
- `/xp list`：查看当前角色的成长标记
- `/advance roll <skill>`：进行一次成长检定并更新技能（必须可审计：前值/掷骰/后值）
- `/downtime start`、`/downtime end`：进入/结束战后流程（KP-only；用于解锁恢复/成长的命令窗口）

##### 细粒度需求（Requirements）

###### R1：成长检定可追溯

- 每次成长必须记录：skill、原始值、掷骰、是否提升、提升后值、依据 event_id

###### R2：治疗/恢复的窗口化

- 恢复（HP/SAN 的缓慢恢复、伤口治疗等）必须限定在 downtime 窗口内（或 KP 显式命令允许），避免战斗中滥用

##### 验收标准（Acceptance Criteria）（M5-D）

- 标记并完成一次技能成长：技能值变化可审计且可在 `/sheet show` 看到
- 进入 downtime 后执行一次恢复/治疗相关落地：过程有记录且不会越权修改其他玩家角色

#### M5-E：端到端短模组验收（调查 + 战斗 + SAN 冲击 +（可选）追逐）

##### 目标

- 用一个 1–3 场景短模组跑通“调查推进 + 规则运转 + 可审计复盘”的完整体验
- 验证 M0–M4 的能力在 M5 的复杂度下仍不崩：门禁/可见性/记忆/检索/引用都稳定

##### 验收标准（Acceptance Criteria）（M5）

- 跑一个含调查 + 战斗 + SAN 冲击的短模组（可选追逐）：
  - 至少发生 1 次规则检索（`/kb rule` 或 `/search scope=rulebook`）并引用 citation
  - 至少发生 1 次模组要素引用（module_ref：NPC/线索/手递物）
  - 至少发生 1 次 SAN 检定并触发后续影响（至少临时疯狂或明确症状标记）
  - 至少发生 1 次远程战斗事件（含 ammo 或护甲其中一项）
- 可复盘：
  - `/recap` 能总结关键事件与 leads
  - `/memory` 能检索回关键承诺/线索
  - 任意一次数值变化（HP/SAN/Luck/技能）都能指回对应 event_id

### 6.7 M6：体验打磨（防卡死与节奏控制）（详细）

#### M6-A：Leads 面板与线索账本体验（玩家不迷路）

##### 目标

- 将 FR-07 的“线索与 leads”从数据字段升级为实际可玩的桌面机制：随时可看、可更新、可关闭
- 在不强推铁路的前提下，稳定维持 2–4 个可行动的推进方向（玩家永远知道“可以做什么”）

##### 新增/细化命令

- `/leads`：查看当前 leads（每条包含：一句话目标、推荐下一步动作、关联线索/对象引用）
- `/leads add "<text>" [link=<clue_id|npc_id|place_id|handout_id>]`：新增一条 lead（KP-only）
- `/leads close <lead_id> [reason]`：关闭/完成某条 lead（KP-only；记录原因）
- `/clues list [scope=me|public|all]`：列出线索账本（按视角过滤）
- `/clues show <clue_id>`：查看线索详情（按视角过滤）

##### 细粒度需求（Requirements）

###### R1：leads 的结构与来源可追溯

- 每条 lead 最小字段：lead_id、text、status（open/closed）、created_from（event_id/module_ref/clue_id 任一）、links（0..n）
- leads 的变更（add/close）必须写入事件日志，可在 `/history` 与 `/recap` 中复盘

###### R2：Leads 数量与质量阈值

- KP 在每轮输出的 `[Next]` 或等价块中，必须给出：
  - 至少 1 个“立即可做”的动作建议（命令级）
  - 若当前 open leads < 2，则必须补齐到 2–4 条（通过 KP-only 机制落地并可审计）

##### 验收标准（Acceptance Criteria）（M6-A）

- 在玩家发散探索 10 轮过程中：
  - `/leads` 始终保持 2–4 条 open leads（允许短暂波动，但必须在 1 轮内恢复）
  - 每条 lead 都能追溯到来源（至少一个 event_id/clue_id/module_ref）

#### M6-B：防卡死与失败前进（Stuck Recovery）

##### 目标

- 当玩家走偏、信息不足或反复失败时，系统能主动提供“失败前进”的新局面，不让游戏陷入长时间停滞
- 失败前进必须可审计（发生了什么代价/计时器推进/线索重投放）

##### 新增/细化命令

- `/hint [level=1|2|3]`：请求提示（玩家命令；不改变状态，只输出建议）
- `/nudge`：推进一小步（KP-only；用于触发失败前进/计时器推进/可移动线索投放）
- `/clock advance <step> [reason]`：推进计时器/压力源（KP-only；与 M0-B7 对齐）

##### 细粒度需求（Requirements）

###### R1：卡死检测的最小启发式

- 至少支持一种卡死检测（实现任选其一，但必须固定）：
  - 连续 N 轮无新线索/无 leads 变化/无 flags 变化
  - 连续 N 次关键检定失败且无状态推进
- 检测到卡死时，KP 必须触发：
  - 生成提示（/hint 风格输出），或
  - 通过 `/nudge` 落地一次失败前进（计时器推进/线索移动/新事件介入）

###### R2：失败前进的落地要求

- 失败前进必须产生“新局面”，并至少写入一种可检索变化：
  - 新线索（clue）
  - 计时器推进（clock）
  - NPC 态度变化/介入
  - 资源损失/受伤/SAN 冲击（必须显式可见、可追溯；必要时要求确认）

##### 验收标准（Acceptance Criteria）（M6-B）

- 人为构造“卡死输入序列”（例如连续 5 轮无效搜索）：
  - KP 能给出 `/hint` 输出（不同 level 更具体，但不剧透 KP-only）
  - 或通过 `/nudge` 落地失败前进，并在 `/recap` 中可追溯

#### M6-C：越界拒绝与命令引导（Better Guardrails UX）

##### 目标

- 保持 NFR-02 的门禁强度，同时让拒绝更友好：告诉用户“为什么拒绝 + 怎么改成可执行的跑团命令”
- 降低新玩家学习成本：把“命令怎么写/下一步怎么做”做成可交互引导

##### 新增/细化命令

- `/help [topic]`：按主题输出（如 combat/chase/sanity/kb/memory）
- `/examples [topic]`：给出 3–5 个常见输入示例（不改变状态）

##### 细粒度需求（Requirements）

###### R1：拒绝模板必须结构稳定

- 对越界/格式错误/权限不足的输入，系统输出至少包含：
  - `[Refusal]`：拒绝原因（简短、明确）
  - `[Allowed]`：允许的替代表达（至少 1 条命令示例）
  - `[Next]`：下一步建议（指向 `/help` 或相关命令）

###### R2：拒绝不改变状态

- 拒绝本身不得隐式推进剧情或修改数值；任何推进必须通过 KP-only 或明确命令链完成

##### 验收标准（Acceptance Criteria）（M6-C）

- 玩家输入跑团外任务：
  - 系统稳定输出拒绝模板，并给出可用命令替代
- 玩家用自然语言试图改数值（例如“我花 5 点幸运”）：
  - 系统拒绝直接改动，并引导到 `/luck 5`

#### M6-D：输出稳定性与信息密度（可读性/节奏）

##### 目标

- KP 输出更易读：固定块、可扫描、默认不冗长；需要时可展开详情
- 降低“长消息墙”对桌面节奏的伤害

##### 新增/细化命令

- `/format <compact|standard|verbose>`：设置输出信息密度（按 viewer 存储偏好）
- `/detail on|off`：切换是否输出审计细节（玩家默认 off；KP 可 on）

##### 细粒度需求（Requirements）

###### R1：每轮输出最小块

- 每轮至少包含三个块（标题可固定）：
  - `[KP]`：叙事与当前反馈
  - `[State]`：关键状态简报（HP/SAN/Luck/异常/当前场景/计时器）按视角过滤
  - `[Next]`：1–3 条下一步动作建议（命令级）

###### R2：可展开而非默认全展开

- 在 compact/standard 下，审计细节（例如骰子展开、计算过程、引用片段）默认折叠为“可查看引用/详情”的提示；通过 `/detail on` 或 show 类命令展开

##### 验收标准（Acceptance Criteria）（M6-D）

- 在 compact 模式跑 10 轮：每轮输出不超过约定上限（例如 1200–1800 字符，具体阈值可实现时再定），但仍包含 `[KP]/[State]/[Next]`
- 在 verbose + detail on：同样输入能看到完整审计细节（骰子展开/引用/计算过程）

#### M6 验收标准（做完即可进入 v1 结束或后续迭代）

- 防卡死：
  - 玩家发散探索不会长期无事可做；当卡死时能通过 hint/nudge/失败前进回到可玩态
- 节奏控制：
  - leads 始终维持 2–4 个可执行方向，且能关闭/完成并形成新的推进
- 门禁体验：
  - 拒绝稳定、可理解、可操作（告诉用户如何改写成可执行命令），且拒绝不改变状态
- 可读性：
  - 输出块稳定，信息密度可配置，玩家默认不被审计细节淹没

### 6.8 M7：Web 控制台（登录 / 管理 / 跑团操作）（详细）

> 目标：在不削弱 TRPG-only 门禁与可审计性的前提下，为非 CLI 用户提供可用的 Web 操作入口；并把“资源管理（脚本/角色卡/跑团）”产品化。

#### M7-A：用户登录与会话（Auth & Session）

##### 目标

- 提供 Web 登录能力，确保资源有清晰归属与访问边界。

##### 细粒度需求（Requirements）

- 登录最小闭环：登录/登出
- 会话鉴权：所有管理类页面必须要求登录；未登录必须跳转登录页
- 权限与视角：
  - Campaign 内至少区分：KP（管理员）与 Player（成员）
  - KP-only 内容不得在 Player 视角渲染（包括列表页、详情页、复盘页）
- 安全最小集：
  - 密码不得明文存储
  - 登录失败/权限拒绝必须写入审计日志（不含敏感明文）

##### 验收标准（Acceptance Criteria）（M7-A）

- 未登录访问任一管理页会被拒绝并跳转登录
- Player 访问 KP-only 页面/字段必被拒绝，且拒绝可审计

#### M7-B：脚本管理（Script Library）

##### 目标

- 在 Web 中管理脚本/场景包资源，支持上传/浏览/校验/版本化（最小可用）。

##### 细粒度需求（Requirements）

- CRUD：创建/上传、列表、查看详情、更新元信息、删除（软删除优先）
- 元信息最小字段：title、system（默认 CoC 7e）、tags、created_by、updated_at、visibility（private/campaign/shared）
- 校验：上传后必须运行“场景包字段校验”，失败则标记为 invalid 且不可绑定到 Campaign
- 版本：同一脚本允许多个版本（version_id），Campaign 绑定到某一版本

##### 验收标准（Acceptance Criteria）（M7-B）

- 上传一个有效脚本可在列表中出现并可查看校验结果
- 上传一个无效脚本会显示错误原因且不能用于开团

#### M7-C：角色卡管理（Character Sheets）

##### 目标

- 在 Web 中创建与维护角色卡，支持导入/导出，并可在 Campaign 中分配给玩家。

##### 细粒度需求（Requirements）

- CRUD：创建、列表、查看、编辑、归档/删除
- 导入/导出：支持 JSON（最小）；字段遵循 M0-C 冻结的角色卡最小字段
- 归属与分配：
  - 角色卡必须有 owner（Account）与可选 campaign_id
  - KP 可将角色卡分配给某 Player（controlled_character_id）
- 防越权：Player 仅能编辑自己拥有/被分配的角色卡（策略可配置，但必须明确）

##### 验收标准（Acceptance Criteria）（M7-C）

- 新建角色卡后可在 Web 编辑，并在同一 Campaign 的 Session 中生效（至少能在状态面板看到关键字段）
- 导出后再导入，关键字段一致（属性/技能/派生数值）

#### M7-D：跑团管理（Campaign Management）

##### 目标

- 将“脚本/玩家/角色卡/多个 Session”组织到一个长期单元（Campaign），便于长期运营与复盘。

##### 细粒度需求（Requirements）

- Campaign CRUD：创建、列表、详情、归档
- 成员管理：邀请/加入（实现可简化为手动添加）、移除（KP-only）
- 绑定资源：为 Campaign 选择脚本（可选绑定具体版本），并关联可用角色卡
- 可见性：Campaign 内的 KP-only 内容不得被非 KP 账号看到

##### 验收标准（Acceptance Criteria）（M7-D）

- KP 创建 Campaign 后可添加 1–3 名 Player，并分配角色卡
- Player 登录后只能看到自己所属 Campaign

#### M7-E：Web 跑团操作台（Play Console）

##### 目标

- 在 Web 页面中完成“自然语言输入 → KP 输出（含 [State]/[Next]）→ 状态/复盘查看”的最小闭环。

##### 细粒度需求（Requirements）

- 会话操作：启动 Session、继续 Session、结束 Session（受权限控制）
- 交互面板：
  - 输入框支持自然语言
  - 提供少量快捷按钮（不要求用户学习命令）：status/recap/help/pause/resume
  - 输出区域必须保留结构化块（至少 `[KP]`、`[State]`、`[Next]`）
- 视角过滤：同一条输出在 Player 视角必须过滤 KP-only 内容
- 状态面板：显示当前 leads、关键 flags、角色关键数值（HP/SAN/Luck）与最近机制事件摘要

##### 验收标准（Acceptance Criteria）（M7-E）

- 在 Web 中完成 ≥5 轮交互推进，状态连续不丢失
- 同一 Session：KP 与 Player 同时在线时，Player 永远看不到 KP-only 片段
